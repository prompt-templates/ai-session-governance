# AIに毎回同じことを説明するのはもうやめよう

Codex、Claude Code、Gemini CLIなどのエージェントワークフローにおいて、プロジェクトを**継続可能・デバッグ可能・追跡可能・整理された状態**に保つための軽量なAI支援開発ガバナンステンプレート。

**[インストール](#インストール)** · **[クイックスタート](#インストール)**

![Overview](ref_doc/overview_infograph.png)

## なぜこれが重要なのか

AIがコーディング、デバッグ、リファクタリング、リリース作業、またはドキュメント更新を支援している場合、本当の問題は通常「モデルが弱い」ことではありません。

本当の問題は、プロジェクトに**持続的なオペレーション記憶がない**ことです。

そこからチームは同じ失敗パターンを繰り返し経験するようになります：

### 課題 1 — 新しいセッションのたびにゼロからスタート
エージェントは現在のベースライン、修正済みの項目、未完了の項目、二度と触れてはいけない項目を把握していません。

### 課題 2 — 修正が古い修正の上に積み重なり続ける
セッションごとにパッチ、メモ、例外、ルールが1つずつ追加され、プロジェクトは変更しやすくなるどころか、変更しにくくなっていきます。

### 課題 3 — バグは局所的に修正されるが、ガバナンスは全体的に悪化する
コードはコンパイルできるかもしれませんが、リポジトリはドリフトしています：
- READMEはある内容を記述
- handoffは別の内容を記述
- logはまた別の内容を記述
- どのバージョンが正しいのか誰にもわからない

### このテンプレートが提供するもの

#### 1) 継続性
新しいセッションごとに強制的なエントリーパスがあります：
- handoffを読む
- logを読む
- 最後に検証された状態から続行する

#### 2) 制御
エージェントは以下のワークフローに従う必要があります：
**PLAN → READ → CHANGE → QC → PERSIST**

これにより、衝動的な編集が減り、トレーサビリティが向上し、変更がより安全になります。

#### 3) 混乱防止ガードレール
このテンプレートは無謀な変更を防ぐだけではありません。
エージェントに自己チェックを強制することで**ガバナンス肥大化**も防ぎます：

- これは本当に新しいルールか？
- それとも古いルールを更新すべきか？
- これは修正か？
- それとも統合パスを実行すべきか？

## インストール

1. **[INIT.md](INIT.md)** を開く → **Raw** をクリック → 全選択 → コピー
2. AIエージェント（Claude Code、Codex、Gemini CLI — いずれでも可）に貼り付ける
3. AIは先にroot安全preflight（絶対パス + リスクチェック + dry-run）を自動実行し、この段階では書き込みしません
4. プロンプトが出たら、次の確認メッセージをそのまま返信します：
   - `INSTALL_ROOT_OK: <absolute_path>`
   - `INSTALL_WRITE_OK`
5. その後、AIが確認済みのプロジェクトルートに5つのガバナンスファイルを作成します

手動セットアップは不要です。AIが全体を自動処理し、既存の `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` の内容もスマートにマージします。
一般的な公開ユーザーにとって、直接使うファイルは `INIT.md` だけで十分です。
このリポジトリ全体を手動でプロジェクト直下へコピーしないでください。`INIT.md` を使って安全にマージしてください。

その後、毎回のAIセッション開始時に以下を使用：

```text
Follow AGENTS.md
```

`Follow AGENTS.md` は標準の短縮表現です。別言語や同義の表現でも動作します。

## クイック操作

自然言語で問題ありません。以下はそのまま貼り付けて使える安定ショートカットです。

### 1) 新しいセッションを開始

```text
Follow AGENTS.md
```

### 2) 同じセッションで作業を継続

```text
Continue from the current state and proceed with PLAN → READ → CHANGE → QC → PERSIST.
```

### 3) セッションを終了して完全に引き継ぐ

```text
Wrap up this session with full closeout and handover.
```

終了時の出力には以下が含まれます：
- `SESSION CLOSEOUT SUMMARY`
- `NEXT SESSION HANDOFF PROMPT (COPY/PASTE)`
- `CLOSEOUT VISUAL CUE`

### 4) 次のセッションをすぐ開始

```text
<Paste the previous "NEXT SESSION HANDOFF PROMPT (COPY/PASTE)" block here, unchanged.>
```

## プラットフォーム設定

`AGENTS.md` は唯一の信頼できるソース（SSOT）です。このテンプレートは薄型ポインターファイル（`CLAUDE.md` と `GEMINI.md`）を同梱しており、すべてのプラットフォームが同一ソースからガバナンスルールを自動検出できます。

| プラットフォーム | ネイティブファイル | テンプレートが提供するもの | 既にファイルがある場合 |
|---|---|---|---|
| **Codex** | `AGENTS.md` | `AGENTS.md`（完全なルール） | ガバナンスセクションを既存ファイルにマージ |
| **Claude Code** | `CLAUDE.md` | ポインターファイル：`@AGENTS.md` | 既存の `CLAUDE.md` の**最上部**に `@AGENTS.md` を追加 |
| **Gemini CLI** | `GEMINI.md` | ポインターファイル：`@./AGENTS.md` | 既存の `GEMINI.md` の**最上部**に `@./AGENTS.md` を追加 |

エージェントが指示を読み込めれば、ガバナンスルール、ワークフロー、dev/テンプレートはすべてのプラットフォームで同一に機能します。

## このテンプレートの違い

ほとんどのAIコーディングプロンプトは以下に焦点を当てています：
- トーン
- フォーマット
- コーディングスタイル
- ツール使用

このテンプレートは長期的な作業にとってより重要なことに焦点を当てています：

### 1) セッション継続性
次のセッションはユーザーの記憶に依存すべきではありません。

### 2) レイヤー分離
エージェントは以下を混同してはなりません：
- プロダクトの振る舞い
- 開発ガバナンス
- 外部プラットフォームの振る舞い
- 環境/ランタイムの問題

### 3) 変更前に読む
「1つの断片を見て、1つの断片を修正する」ワークフローは許可されません。

### 4) 追加前に統合
ルールと例外を際限なく積み上げることは許可されません。

### 5) セッションを正しく終了する
エージェントは離開前にhandoff/logを更新する必要があります。

## 3つのシナリオ

### シナリオ 1 — 毎日AIで出荷する個人開発者
日々の進捗は速いが、セッション間でコンテキストが途切れます。
このテンプレートは各セッションに安定した再エントリーポイントを提供し、繰り返しの説明を削減します。
また、「小さな修正」が長期的なガバナンスの混乱に発展するのを防ぎます。
プロダクトビルダー、インディーハッカー、技術系創業者に適しています。

### シナリオ 2 — 1つのリポジトリ、複数のAIエージェント
Codexがコードを処理し、Claudeがドキュメントをレビューし、Geminiがインフラのデバッグを支援します。
共通のhandoffとセッションログがなければ、各エージェントは異なる現実から作業します。
このテンプレートは共有オペレーション記憶と一貫したセッションID標準を作成します。
マルチエージェントワークフローに適しています。

### シナリオ 3 — 既存プロジェクトがすでに混乱している
リポジトリは動作するが、修正のたびにルールが長くなり、ドキュメントがうるさくなり、リリースがリスキーになります。
このテンプレートは蓄積防止の規律を追加します：まず検索、まず統合、古いルールを廃止。
ガバナンスドリフトを減らすように設計されており、ガバナンスを増やすだけではありません。
完全な書き換えなしにクリーンアップが必要な長期リポジトリに適しています。

## よくある質問

### 1) これは大規模プロジェクト専用ですか？
いいえ。
小規模プロジェクトはセッション継続性からすぐに恩恵を受けます。
大規模プロジェクトはドリフトが時間とともに複合的に加速するため、さらに大きな恩恵を受けます。

### 2) 初日から `PROJECT_MASTER_SPEC.md` が必要ですか？
必ずしもそうではありません。
小規模リポジトリでは `AGENTS.md` + `SESSION_HANDOFF.md` + `SESSION_LOG.md` で十分です。
ルール、リリース基準、またはワークフローが非公式な管理では大きすぎるようになったら、master specを追加してください。

### 3) これはコーディング標準ですか？
主にそうではありません。
これはAIがリポジトリ内でどのように作業するかの**ガバナンス標準**です：
- どのように読むか
- どのように変更するか
- どのように検証するか
- どのように引き継ぐか

### 4) なぜすべてを1つの大きなpromptにまとめないのですか？
大きなpromptは劣化するからです。
更新しにくく、信頼しにくく、次のセッションで一貫して適用しにくくなります。
このテンプレートは安定したガバナンスをセッションごとの状態から分離します。

### 5) これはエージェントを遅くしますか？
各セッションの開始時に少しだけ遅くなります。
通常、繰り返しの説明、誤診、重複修正、古いドキュメントによって失われる時間よりもはるかに少ないです。
ポイントは合計の手戻りを減らすことであり、速いが忘れっぽい1回のターンを最適化することではありません。

### 6) 「追加前に統合」はどんな問題を解決しますか？
ガバナンス肥大化を防ぎます。
このルールがなければ、すべての修正が追加メモに、すべてのインシデントが追加SOPに、すべてのセッションがより多くの散らかりを残します。
このテンプレートはエージェントにルールを単に追加するのではなく、マージすべきかどうかを考えさせます。

### 7) 問題がいつ永久ルールになるべきですか？
繰り返し発生する、高影響の、リリースをブロックする、リスクのある、またはシステム的な場合のみです。
問題が小さくて局所的であれば、根本原因を修正し、リグレッションを追加し、ログを更新して、先に進んでください。
すべてのミスが新しい法律に値するわけではありません。

### 8) これはデバッグに役立ちますか、それともドキュメントだけですか？
両方です。
まず問題トリアージを強制することでデバッグを改善します：
- コードの問題？
- 設定の問題？
- ランタイムの問題？
- 外部プラットフォームの問題？
- 古いドキュメント？
これにより「間違った方向の修正」デバッグが減少します。

### 9) プロジェクトにすでにREADME、docs、内部ルールがある場合は？
それらを保持してください。
このテンプレートは既存のプロジェクト資料と統合するためのものであり、有用なコンテキストを盲目的に置き換えるものではありません。
唯一の厳格な要件は：信頼できるエントリーパスと永続的なセッション記録です。

### 10) このテンプレートが防止するように設計されている最大のミスは何ですか？
静かなミスです：
まだ「動く」プロジェクトが、AIセッションのたびに理解しにくく、変更しにくく、信頼しにくくなっていくこと。
このテンプレートはその緩やかな劣化を早期に止めるために存在します。

## この repo のソース構成

```text
<PROJECT_ROOT>/
├─ INIT.md                ← bootstrap prompt（公開エントリー）
├─ AGENTS.md              ← ガバナンスルール（SSOT）
├─ CLAUDE.md              ← Claude Code用ポインター
├─ GEMINI.md              ← Gemini CLI用ポインター
└─ dev/
   ├─ SESSION_HANDOFF.md
   ├─ SESSION_LOG.md
   └─ PROJECT_MASTER_SPEC.md   # オプション
```

### コアファイル

* `INIT.md` — プロジェクト内にガバナンスファイルを作成/マージするためのbootstrap prompt（多くのユーザーの主要エントリー）
* `AGENTS.md` — リポジトリ内でのAI作業の常設オペレーションルール（唯一の信頼できるソース）
* `CLAUDE.md` — Claude Codeの自動検出を `AGENTS.md` にブリッジするポインターファイル
* `GEMINI.md` — Gemini CLIの自動検出を `AGENTS.md` にブリッジするポインターファイル
* `dev/SESSION_HANDOFF.md` — 現在のベースライン、ブロッカー、開始チェックリスト、最後の検証状態
* `dev/SESSION_LOG.md` — セッションごとの履歴、修正、検証、次の優先事項
* `dev/PROJECT_MASTER_SPEC.md` — 大規模または複雑なプロジェクト向けのオプション長期権威仕様

## このテンプレートの背後にあるガバナンス原則

このリポジトリはいくつかの譲れない原則を中心に構築されています：

1. **変更前に読む**
2. **デバッグ前にトリアージ**
3. **追加前に統合**
4. **完了を主張する前に検証**
5. **離れる前に永続化**

セッション終了時、エージェントは実際の現在状態から動的生成したコピペ可能な引き継ぎ指示を出力する必要があります（固定文は禁止）。

この5つが維持されれば、AIセッションは時間が経っても使用可能であり続けます。

## 対象者

このテンプレートは以下の方に最も役立ちます：

* 毎週AIでコーディングする
* 複数のモデルやエージェントツールを切り替える

* 持続的で追跡可能な進捗を重視する

心当たりがあれば、このテンプレートは実用的な出発点となることを目指しています。
